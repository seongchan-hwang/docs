# 기술 보고서 작성 가이드

이 문서는 AI Agent가 `tech-report-template.md`를 채울 때 참조하는 섹션별 작성 지시이다.

---

## 파일 규칙

- **저장 위치**: `TECHNICAL_REPORT/` (프로젝트 루트) 하위에 저장한다
- **파일명 형식**: `{INCR_NUMBER}-{title}-{yyyyMMdd}.md`
  - `INCR_NUMBER`: 디렉토리 내 기존 파일의 최대 번호 + 1 (첫 파일은 `001`)
  - `title`: PR 내용을 요약한 kebab-case 영문 제목
  - `yyyyMMdd`: 보고서 생성일
  - 예시: `001-agent-builder-structured-output-20260403.md`

---

## 헤더

- **작성일**: 보고서 생성 시점의 날짜 (YYYY-MM-DD)
- **작성자**: 보고서를 생성한 AI 에이전트 식별자
- **위험도**: PR 전체의 위험 수준을 평가한다
  - **낮음**: 문서, 테스트, 설정 변경 등 런타임 영향 없음
  - **보통**: 기존 로직 수정이 있으나 영향 범위가 제한적
  - **높음**: 인증/권한, 데이터 모델, 외부 API 계약 등 핵심 경로 변경

---

## 요약

다음 세 가지 질문에 답하는 형태로 작성한다:

1. **무엇을 왜 변경했는가?** — 변경의 동기와 목적
2. **어떤 접근 방식을 선택했고, 핵심 트레이드오프는?** — 기술적 선택의 근거
3. **리뷰어가 반드시 확인해야 할 부분은?** — 주의 깊게 봐야 할 파일이나 로직

<example>
Agent Builder의 LLM 응답 처리를 자유 텍스트 파싱에서 tool_use structured output으로 전환했다. 자유 텍스트 파싱은 응답 형식이 불안정하여 프로덕션에서 파싱 실패가 빈번했고, 이를 해결하기 위해 Anthropic API의 tool_use 기능을 활용하여 스키마 수준에서 출력 형식을 강제했다.

핵심 트레이드오프는 프롬프트 복잡도 증가 vs 파싱 안정성이며, 파싱 안정성을 우선했다.

리뷰어는 `backend/app/services/agent_builder.py`의 tool definition 스키마와 `parse_tool_response()` 함수를 중점적으로 확인해야 한다.
</example>

## 1. 문제정의

### 1.1 배경
이 변경이 필요해진 맥락을 서술한다. 관련 이슈/티켓 번호가 있으면 포함한다.

### 1.2 기존 문제
변경 전 상태에서 어떤 문제가 있었는지 구체적으로 기술한다. 재현 조건이나 증상을 포함한다.

### 1.3 위험도 평가
이 문제를 방치했을 때의 위험과, 이번 변경 자체가 도입하는 위험을 구분하여 서술한다.

### 1.4 영향 범위
이 변경이 직접적으로 영향을 미치는 서비스, 모듈, API, 사용자 그룹을 테이블에 나열한다. 간접적 영향이 있으면 비고에 명시한다.

---

## 2. 기술검토

### 2.1 보안 분석
검토 항목을 나열하고, 각 항목의 결과를 기술한다. 평가는 3단계로 판정한다:
- **적합**: 보안 이슈 없음
- **조건부 적합**: 즉각적 위험은 없으나 주의사항이나 개선 권고가 있음
- **부적합**: 보안 취약점이 존재하여 병합 전 수정 필요

### 2.2 성능 분석
변경이 성능에 미치는 영향을 분석한다. 해당 사항이 없으면 "성능 영향 없음"으로 판정하고 근거를 명시한다. 평가는 보안 분석과 동일한 3단계를 사용한다.

### 2.3 코드 품질
테스트 커버리지, 문서화, 코드 일관성을 각각 평가한다. 테이블의 "비고" 열에 근거나 구체적 수치를 기입한다.

---

## 3. 기술적 결정

**"왜 이 방식을 선택했는가"**를 다루는 섹션이다. 설계 판단과 트레이드오프를 기록하여 향후 동일한 결정을 재논의하지 않도록 한다.

각 결정마다 다음 구조로 작성한다:
- **결정**: 무엇을 결정했는가
- **대안**: 고려한 다른 방식은 무엇이었는가
- **근거**: 왜 이 방식이 대안보다 나은가 (트레이드오프 포함)

구현 방법이나 코드 레벨 상세는 섹션 4에서 다룬다.

<example>
**결정**: LLM 응답 파싱을 regex 기반에서 tool_use structured output으로 전환

**대안**:
- (A) 기존 regex 파싱 유지 + 재시도 로직 강화
- (B) JSON mode 사용
- (C) tool_use structured output 사용

**근거**: (A)는 LLM 출력 변동성을 근본적으로 해결하지 못한다. (B)는 스키마 강제가 불가능하여 필드 누락 위험이 남는다. (C)는 API 레벨에서 스키마를 강제하므로 파싱 실패를 원천 차단한다. 프롬프트 복잡도가 증가하나 런타임 안정성이 우선이다.
</example>

---

## 4. 구현 상세

**"실제로 무엇을 어떻게 바꿨는가"**를 다루는 섹션이다. 코드 레벨의 변경 사항을 기술한다:
- 추가/수정/삭제된 주요 파일과 그 역할
- 데이터 흐름이나 호출 구조의 변경
- 필요시 코드 스니펫으로 핵심 변경을 보여준다

선택의 근거는 섹션 3에서 다룬다.

## 7. 변경 요약

### 통계
git diff 기반으로 정확한 수치를 테이블에 기입한다. 추정치가 아닌 실측값만 사용한다.

### 카테고리 분류
이 PR이 해당하는 카테고리를 체크한다. 하나의 PR이 여러 카테고리에 해당할 수 있다. "비고" 열에 해당 이유를 간략히 적는다.

### 관련 커밋
PR에 포함된 모든 커밋을 테이블에 나열한다. 유형은 conventional commit 분류를 따른다 (feat, fix, refactor, docs, test, chore 등).

---

## 8. 후속 조치

### 완료됨
이 PR로 해결된 항목을 나열한다.

### 테스트 필요
병합 후 추가로 검증이 필요한 시나리오를 나열한다.

### 모니터링
배포 후 관찰해야 할 지표나 로그를 명시한다.

### 롤백 계획
문제 발생 시 롤백 방법을 기술한다. 단순 revert로 충분한지, 데이터 마이그레이션 롤백이 필요한지 등을 명시한다.

### 향후 개선 사항
이번 PR 범위에서 의도적으로 제외한 개선 사항을 나열한다.
